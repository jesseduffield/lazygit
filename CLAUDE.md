# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

**Before doing anything else, read AGENTS.md and follow it.** It is the source of truth for
workflow: which `just` recipes to run, when and how to commit (fine-grained, `fixup!`-based),
the bug-demonstration (`EXPECTED`/`ACTUAL`) pattern, integration-test conventions, and the rule
not to touch `docs/`. The notes below cover only the codebase's architecture; they do not
restate AGENTS.md.

## What this is

Lazygit is a terminal UI for git, written in Go. `main.go` calls `pkg/app`, which sets up
config/logging and starts the GUI. The GUI runs on a vendored fork of the `gocui` library
(`vendor/github.com/jesseduffield/gocui`), which owns the event loop, keypress handling, and
rendering.

## Commands

All routine commands go through `just` (see `just --list` and AGENTS.md). The ones you'll use most:

- `just build` / `just run` — build / build-and-run the binary.
- `just unit-test` — `go test ./... -short`.
- `just e2e [<name>]` — run integration tests headlessly; `just e2e commit/new_branch` runs one
  (the name is its path under `pkg/integration/tests/`). `just e2e-cli <name>` runs one in a
  visible UI; `just e2e-tui` browses them.
- `just generate` — regenerate auto-generated files (integration-test list, keybinding
  cheatsheets, `docs-master/Config.md`, `schema-master/config.json`). Run after adding/renaming a
  test, changing keybindings, or editing `userConfig` field doc comments; CI fails if stale.
- `just format` (gofumpt) and `just lint` (golangci-lint) before committing.

## Architecture

`docs/dev/Codebase_Guide.md` is the authoritative tour — read it for anything non-trivial. The
essentials:

The GUI follows a Context / Controller / Helper separation that has been progressively carved out
of a legacy `Gui` "god struct" (which still holds leftover state and some keybindings in
`pkg/gui/keybindings.go`):

- **Context** (`pkg/gui/context/`) — one per view (branches, commits, files, …). Holds
  view-specific state (its ViewModel) and writes content to its view. Set up in `context/setup.go`.
- **Controller** (`pkg/gui/controllers/`) — defines keybindings and their handler actions. One
  controller can serve many contexts and one context can have many controllers. Wired to contexts
  in `pkg/gui/controllers.go`.
- **Helper** (`pkg/gui/controllers/helpers/`) — shared logic two controllers both need.
  Controllers cannot call each other's methods, so common code is extracted into a helper.

**Dependency direction (strict):** controllers → helpers → contexts → views. Views never reference
contexts/controllers/helpers; contexts only reference views. Respect this when deciding where new
code goes. Rule of thumb: code starts in a controller and moves to a helper once a second
controller needs it.

Other layers:

- **Git layer** (`pkg/commands/git_commands/`) — the *only* place that shells out to git (e.g.
  `Checkout` runs `git checkout`). `git.go` assembles the command structs; `oscommands/` runs
  processes; loaders (`*_loader.go`) read git state into `pkg/commands/models/` structs.
- **Presentation** (`pkg/gui/presentation/`) — renders models into the text shown in views.
- **`Common` struct** (`pkg/common`) — the dependency bag (logger, i18n, UserConfig) threaded
  through nearly every struct as a field named `c`; helpers are reached via `self.c.Helpers.X`.
- **Refresh** (`helpers/refresh_helper.go`) — after an action, models are re-loaded from git here
  (e.g. reload branches after a pull) and views re-rendered.

**Threads:** the event loop is single-threaded and redraws via `layout` (`pkg/gui/layout.go`) after
each event. Run blocking work off the UI thread with `self.c.OnWorker(fn)`, and hop back with
`self.c.OnUIThread(fn)`.

## Config

User config is defined in `pkg/config/user_config.go`. It can be hot-reloaded while lazygit runs,
so read it live through `self.c.UserConfig()` rather than caching a value. The `userConfig` field
doc comments are the source for `docs-master/Config.md` and the JSON schema (regenerated by
`just generate`) — AGENTS.md has the exact rules for writing those comments (notably: don't
hard-wrap them).

## Integration tests

Each test (`pkg/integration/tests/<dir>/<name>.go`) has a `shell` setup step that builds a repo and
a run step that drives a real lazygit session via `t` and asserts on it. They're registered in the
generated `test_list.go` (`just generate`). See `pkg/integration/README.md` and
`docs/dev/Integration_Tests.md`, and AGENTS.md for the fluent `t.Views().<View>()` style required.
